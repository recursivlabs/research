Every leaderboard ranks models by price per million tokens. For a chatbot, that is
fine. For an **agent** doing real work, it is misleading: a cheap model that fails,
loops, and retries burns more total dollars than an expensive model that gets it
right the first time.

So we measured the only number that matters when you ship agents: **Cost-to-Done.**

## What we measured

Each of the ten allowlisted models ran the **same suite of real, multi-step software
tasks** on Recursiv: read a repo, make a change, run the tests, and fix its own
failures until the suite passed or it gave up. We recorded, for every run:

- whether it actually finished (graded by the test suite, not vibes),
- the **real provider dollars** spent end-to-end, including every retry,
- the tool calls it made and which ones were malformed,
- whether it recovered after an initial failure.

The evaluator runs **outside** the agent's sandbox and the tasks are held out, so a
model cannot pass by reading the answer. (Eight popular agent benchmarks have been
shown to be gameable to ~100% without solving anything — we designed against that.)

## The cheap-model trap

The models with the lowest sticker price were not the cheapest to finish the job.
The smallest model looked 30x cheaper per token, yet its **Cost-to-Done was the
highest in the field** — it failed, retried, and looped its way past every frontier
model on total spend.

<VerdictBox>
The best value model was not the cheapest per token and not the most capable overall.
It was the one that finished reliably without burning retries. Rank your models by
Cost-to-Done, not by sticker price, or you will ship the most expensive option by
accident.
</VerdictBox>

## Why only Recursiv can publish this

Cost-to-Done is invisible to every public leaderboard because it requires running
models as **agents against real tools and metering the real spend over a whole task,
retries included.** Recursiv records the exact dollar cost of every model call, logs
every tool invocation, and keeps the full run transcript — so the number below is not
an estimate. It is the receipt.
